A mental model for Linux file, hard and soft links

I always felt bothered about my superficial understanding of inode, hard and soft links in Linux. Here I attempt to structure my learnings into a simplified mental model. Corrections are welcome.

What makes a file in Linux?

At the storage level, a file is, of course, a data block. There is more to it than just that, but I won't delve deep into it.

At the filesystem level, a file is represented by an abstraction called inode. Think of inode as a data structure that stores metadata about the underlying data block of the file.

An inode is made accessible with a filename, which is, crudely put, a pair of absolute filename and a pointer to inode..

Roughly, file = filename + inode + data block.

So, when you say a file, you may be referring to any or all of these components.

Hard links

Now, the data block of a file is represented by exactly one inode at the filesystem level. But, if we think of filenames as labels slapped on inodes, then we can refer to the same inode with different labels.

In the above figure, filename1 and filename2 point to the same inode. These two are hard links. A hard link connects a filename with an inode.

The above figure can be interpreted in several ways:

  1. Given that a filename is represented as a pair of its absolute name and a pointer to an inode, a hard link is maintained in the filename's implementation data structure.

  2. Extending the above point, a hard link points a filename to an inode. Thus, the filename is aware of the inode, but the inode is not aware of the filename.

  3. At this level of abstraction, is it possible to tell which one of them originally created the file? I think not. Hard links are supposed to be equal in status to each other.

  4. As a corollary to the above point, there is always at least one hard link attached to a file. When you create a file, say, with touch filename, you create a hard link between filename and the file's inode.

Deleting a hard link

Refer to the figure above. If you do rm /tmp/filename1, what should happen from a sane perspective?

Should the entire filename1 + inode + data block combo be removed? Then what happens to /home/bhoot/filename2?

The right way would be to preserve the integrity of /home/bhoot/filename2 while deleting /tmp/filename1.

That's what happens when you delete a file - the (hard) link between that specific filename and the inode is removed.

Similarly, mv f1 f2 creates a new hard link f2 & deletes the old hard link f1.

What happens when all the hard links to a file are deleted? Now, there are no filenames in the filesystem that refer to that data block anymore. Thus, the inode is marked for deletion, and the associated data block is orphaned.

Soft links

Take a look at the following figure.

The above figure shows a soft link in blue colour (its blue, right?) pointing from the link file filename3 to the target file filename2. A soft link or a symbolic link points a link file to a target filename. The link file, which stores the connection, points to the target filename.

A soft link links a link file to a target filename. This is in contrast to a hard link, which links a filename to an inode.

In the man page of the POSIX ln, the syntax description goes as: ln [-s] source target, where the source is the file to be linked to, while the target is the new link file to be created. These designations may make sense within the specific context of creating a soft link. But, once a soft link is already created, calling it target doesn't make sense.

I prefer the terms used in the man page of the GNU ln: ln [-s] target link_name, where target is the file to be pointed towards, while link_name is the new link file.

A soft link file is still a file

As we saw in the What is a file in Linux? section, a filename is always linked to an inode + data block. This is true in case of a soft link, too. In the above figure, the soft link file filename3, which points to filename2, also has its own inode & a data block.

You can check the inode number associated with a filename with ls -i command. Two hard link files pointing to the same inode will print the same inode number, while a soft link file and its target file will print different inode numbers.

# create a regular file
$ echo "HELLO WORLD HOW ARE YOU?" > file.txt

# create a hard link
$ ln file.txt hardlink.txt

# both hard links refer to the same inode
$ ls -i file.txt hardlink.txt 
17751620 file.txt  17751620 hardlink.txt

# create a soft link
$ ln -s file.txt softlink.txt

# inode of soft link file is different from the target file
$ ls -i file.txt softlink.txt 
17751620 file.txt  17751663 softlink.txt

Contents of a soft link file

A hard link relationship is stored inside the filename's data structure, as a pointer to the inode.

How is a soft link relationship stored? In the data block of the soft link file. The content of a soft link is the filename of the target file it points to.

If a soft link file is created in the same directory as the target file, then only the basename of the target file is stored as the content of the soft link file. Otherwise, the absolute filename of the target file is stored.

10th of November, 2024: Kosayoda on lobste.rs politely pointed out that my struck-out claim above is bullshit. After a few tests, I concur.

Check out the full explanation, but a concise summary is: ln -s some-file.txt link-file.txt simply creates a link-file.txt file, the content of which is some-file.txt.

There are 2 ways to verify this:

  1. readlink filename3 prints the content of the soft link filename3, which is the name of the target filename.

    $ readlink filename3
/home/bhoot/filename2

  2. stat --format="%s" filename3 prints the size of filename3, which is equal to the number of characters in the name of /home/bhoot/filename2.

    $ stat --format="%s" filename3
21

I/O operations on a soft link file

Copying a soft link file

What happens when you copy a soft link file? Same thing as when a regular file is copied. Or so I thought.

# create a regular file
$ echo "HELLO WORLD HOW ARE YOU?" > file.txt

# check the content size of target file
$ stat --format="%s" file.txt
25

# create a soft link
$ ln -s file.txt softlink.txt

# check the size of the soft link file
$ stat --format="%s" softlink.txt
8

# copy the soft link file
$ cp softlink.txt softlink2.txt

# check the size of the new soft link file, which should be equal to the soft link file
$ stat --format="%s" softlink2.txt
25 # WAT?

What happened there? The cp command simply copied the target file instead of the link file.

My guess is that when those I/O operations, which involve acting on the data block of a file, are ran on a soft link file, they are actually executed on the target file that the soft link file is pointed to.

Thus, cp softlink.txt softlink2.txt would actually copy the target file. Similarly, cat softlink.txt would print the content of the target file. But, mv, which does not need to act on the underlying data block of a file, would just operate on the soft link file itself.

Moving a soft link file

What happens when you move a soft link file? This is more predictable. Thankfully, the same as what happens when a regular file is moved - The old filename entry is removed. The new filename entry is created, while the old filename entry is removed. Because the new filename still points to the same old inode + data block, it also points to the same target file.

Moving the target file

A more interesting question worth pondering: what happens when the target file is moved or deleted? Let's go with a case of severe deletion.

A soft link file depends on its content to determine which file it points to. Even when the target file is deleted, the content of soft link file doesn't change, i.e., it still keeps pointing to the now deleted target file. With its purpose existing no more, the soft link thus becomes a dangling, dead link.

In closing

From the perspective of structural existence, hard links actually look softer than soft links to me.
